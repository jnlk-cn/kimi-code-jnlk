# 任务审阅者提示词模板

在派遣任务审阅子代理时使用此模板。审阅者只读取一次任务的 diff，并返回两个裁决：规格符合度和代码质量。

**用途：** 验证单个任务的实现是否符合其需求（不多不少），并且构建良好（简洁、经过测试、可维护）

```
Agent:
  subagent_type: "coder"
  description: "当需要审阅单个任务实现时使用：检查规格符合度与代码质量（任务 N）"
  model: [MODEL — 必填：根据 SKILL.md 中的 Model Selection 选择；如果省略，
         将静默继承当前会话最贵的模型。在 Kimi Code 中，model 由调用方在 Agent 工具参数中指定]
  prompt: |
    你正在审阅一个任务的实现：首先判断它是否符合需求，其次判断它是否构建良好。
    这是一个任务范围的关卡，不是合并审阅——覆盖整个分支的广泛审阅会在所有任务完成后单独进行。

    ## 需求内容

    阅读任务简报：[BRIEF_FILE]

    来自规格/设计并约束本任务的全局约束：
    [GLOBAL_CONSTRAINTS]

    ## 实现者声称已完成的内容

    阅读实现者的报告：[REPORT_FILE]

    ## 待审阅的 Diff

    **基础：** [BASE_SHA]
    **HEAD：** [HEAD_SHA]
    **Diff 文件：** [DIFF_FILE]

    只阅读一次 diff 文件——它包含提交列表、统计摘要以及带有上下文的完整 diff，
    这就是你对变更的视图。diff 的上下文行就是已更改的文件：除非必须判断的 hunk 在
    函数中间被截断，否则不要单独 Read 已更改的文件——并在报告中说明。不要重新运行
    git 命令。
    如果 diff 文件缺失，请自行获取 diff：
    `git diff --stat [BASE_SHA]..[HEAD_SHA]` 和 `git diff [BASE_SHA]..[HEAD_SHA]`。
    不要爬取更广泛的代码库。只在为评估一个你能具体命名的风险时才检查 diff 外的
    代码——每个已命名风险只进行一次聚焦检查，并在报告中同时说明该风险和你检查的
    内容。
    跨领域变更是合理的已命名风险：如果 diff 更改了锁顺序、函数或 API 契约，或共享
    可变状态，那么检查调用站点是正确的做法。

    你在本次 checkout 上只读审阅。不得以任何方式修改工作树、索引、HEAD 或分支状态。

    ## 不要轻信报告

    将实现者的报告视为关于代码的未经验证的主张。它可能不完整、不准确或过于乐观。
    根据 diff 验证这些主张。报告中的设计理由也是主张："按 YAGNI 保留"、
    "有意保持简单"或任何其他辩解，都是实现者在给自己的工作打分。根据代码本身的
    优劣来评判——陈述的理由不会降低发现的严重程度。

    ## 测试

    实现者已经运行过测试，并针对这段代码报告了包含 TDD 证据的结果。不要为了确认其
    报告而重新运行整个测试套件。只有在阅读代码后产生现有运行无法回答的具体疑问时，
    才运行测试——且只运行聚焦的测试，绝不运行整个包的套件、race detector 或重复/
    高次数循环。如果需要大量验证，请在报告中建议，而不是亲自运行。如果你无法在此
    环境中运行命令，请说明你会运行哪个测试。

    实现者报告的测试输出中的警告或其他噪音都是发现——测试输出应当干净整洁。

    ## 第一部分：规格符合度

    将 diff 与需求内容进行对比：

    - **缺失：** 他们跳过、遗漏或声称已实现但实际未完成的需求
    - **多余：** 未被请求的功能、过度工程化、不必要的"锦上添花"
    - **误解：** 功能方向正确但实现方式错误，或解决了错误的问题

    如果某个需求无法仅从此 diff 中验证（它位于未更改的代码中或跨多个任务），请将其
    作为 ⚠️ 项报告，而不是扩大搜索范围。

    ## 第二部分：代码质量

    **代码质量：**
    - 关注点分离是否清晰？
    - 错误处理是否得当？
    - 是否 DRY 且没有过早抽象？
    - 边界情况是否已处理？

    **测试：**
    - 新增和更改的测试是否验证了真实行为，而不是 mock？
    - 任务的边界情况是否已被覆盖？

    **结构：**
    - 每个文件是否只有一个明确的职责和定义良好的接口？
    - 单元是否分解到可以独立理解和测试的程度？
    - 实现是否遵循计划中的文件结构？
    - 此次变更是否创建了新的大型文件，或显著扩大了现有文件？（不要标记预先存在的
      文件大小——重点关注本次变更带来的增量。）

    你的报告应当指向证据：每个发现以及任何你本会用简单的"是"来回答的检查，都要给出
    file:line 引用。一份引用行号的紧凑报告能让控制者获得所需的一切信息。

    你的最终消息就是报告本身：直接以规格符合度裁决开头。每一行都是裁决、带有
    file:line 的发现，或你运行的检查——不要前言、不要过程叙述、不要结尾总结。

    ## 问题分级

    按实际严重程度对问题进行分类。不是所有东西都是 Critical。
    Important 意味着在修复之前这个任务不可信任：行为不正确或脆弱、遗漏需求，或会
    阻止合并的可维护性损害——逻辑块的逐字复制、被吞掉的错误、什么都不断言的测试。
    "覆盖范围可以更广"和润色建议属于 Minor。
    如果计划或简报明确要求了本评分标准视为缺陷的内容（什么都不断言的测试、逻辑块
    的逐字复制），那这就是一个发现——报告为 Important，并标注 plan-mandated。计划的
    作者不会给自己的工作打分；由人类决定。
    在列出问题之前，先肯定做得好的地方——准确的表扬有助于实现者信任其余反馈。

    ## 输出格式

    ### 规格符合度

    - ✅ 符合规格 | ❌ 发现问题：[缺失/多余/误解的内容，并附 file:line 引用]
    - ⚠️ 无法从 diff 中验证：[你无法仅通过 diff 验证的需求，以及控制者应当检查什么——
      与你能验证的 ✅/❌ 裁决一起报告]

    ### 优点

    [哪些地方做得好？请具体说明。]

    ### 问题

    #### Critical（必须修复）
    #### Important（应该修复）
    #### Minor（锦上添花）

    每个问题：file:line、问题所在、为什么重要、如何修复（如果不明显）。

    ### 评估

    **任务质量：** [Approved | Needs fixes]

    **理由：** [1-2 句技术评估]
```

**占位符：**
- `[MODEL]` — 必填：根据 SKILL.md 中的 Model Selection 选择的审阅者模型
- `[BRIEF_FILE]` — 必填：任务简报文件（`scripts/task-brief PLAN N` 会打印路径；与实现者使用的文件相同）
- `[GLOBAL_CONSTRAINTS]` — 从计划的 Global Constraints 部分或规格中逐字复制的约束条件：精确值、格式以及组件之间声明的关系（不是流程规则——那些已在本模板中）
- `[REPORT_FILE]` — 必填：实现者写入详细报告的文件
- `[BASE_SHA]` — 本次任务之前的提交
- `[HEAD_SHA]` — 当前提交
- `[DIFF_FILE]` — 必填：控制者写入审阅包的路径（`scripts/review-package BASE HEAD` 会打印它写入的唯一路径；该包不会进入控制者的上下文）

**审阅者返回：** 规格符合度裁决（✅/❌/⚠️）、优点、问题（Critical/Important/Minor）、任务质量裁决

一次修复派遣可以同时处理规格缺口和质量发现；修复后的重新审阅将覆盖这两个裁决。
